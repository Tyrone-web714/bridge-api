| ODR-002 | What is the approved mobile GitHub repository? | Create or use a dedicated private GitHub repository for `C:\dev\tsr-mobile`. | Keep local only; place mobile inside backend repo; create monorepo now. | Local-only mobile source is a release and recovery risk; monorepo conversion is out of scope for Phase 0. | Phase 0 | Reclassified as Engineering Decision. Rationale: Staging is now approved by ODR-014; the remaining work is provisioning or confirming the concrete restore target and validating restore operations. | Operational Verification |
| ODR-005 | What backup retention is required for pilot? | Minimum 7 daily and 4 weekly backups, with longer retention for receipts, signatures, photos, GPS trails, and audit logs. | Shorter pilot retention; longer compliance retention. | Retention affects storage cost, legal exposure, and recovery windows. | Phase 0 | Reclassified as Operational Verification. Rationale: ODR-013 approved category-based retention; remaining work is verifying provider settings and aligning pilot backup retention with that policy. | Operational Verification |
| ODR-006 | Are company driver IDs globally unique or Organization-scoped? | Use two identifiers: a globally unique internal driver ID using UUID or equivalent for the permanent platform database identity, and an Organization-scoped `company_driver_number` for employer operational identification. Enforce uniqueness on `(organization_id, company_driver_number)`. | Require company driver numbers to be globally unique; use driver names as identifiers; use only internal IDs in mobile/operator workflows. | Organization-scoped company driver numbers support future tenants while preserving operational mobile/API compatibility; backend relationships must resolve to the permanent internal driver ID. | Phase 1 | Approved. Do not require company driver numbers to be globally unique. Do not use driver names as identifiers. Mobile app and APIs may continue displaying or accepting company driver number where operationally appropriate. | Decided |
| ODR-007 | What exact field name should represent the company driver/employee number? | Use `company_driver_number` for drivers and `company_employee_id` for non-driver staff. | Reuse `driver_id` for all cases; use only internal IDs. | Clear naming prevents confusing staff employee identity with driver route assignment identity. | Phase 1 | Reclassified as Engineering Decision. Rationale: ODR-006 approved the identity model; exact column/API field naming is an implementation detail that must preserve that model. | Engineering Decision |
| ODR-008 | Which role hierarchy is approved for Truck-Safe Routing? | Use five role types: Platform Admin, Organization Admin, Supervisor, Driver, and Warehouse Employee. Enforce least privilege through explicit permissions and Organization context, not role names alone. | Add separate role types for Dispatcher, Maintenance staff, Customer service, Auditor, External inspector, Regional Manager, Depot Manager, Safety Manager, or Executive Viewer; use one shared admin role; rely on page-level ad hoc roles. | A fixed five-role hierarchy keeps the permission model understandable while still allowing specialized functions through permissions. No Organization-level role may access another Organization's private data. | Phase 3-4 | Approved. Platform Admin manages SaaS-wide controls; Organization Admin manages one Organization; Supervisor manages assigned Organization operations; Driver accesses assigned personal operational data; Warehouse Employee performs warehouse-specific workflows. Unsupported functions must be permissions under approved roles, not new role types. | Decided |
| ODR-009 | Which API endpoints are allowed to remain public? | Adopt a "Private by Default" API security model. Public endpoints are limited to explicitly approved non-sensitive platform functions such as health check, readiness check, authentication bootstrap, password reset initiation, public documentation if enabled, and public status page if enabled. All Organization operational data requires authentication and Organization context. | Leave current public-or-inline behavior; protect every endpoint including health/readiness; allow clients to pass arbitrary Organization identifiers. | Private by Default reduces tenant-isolation risk before migration. Public writes and Organization data exposure are not acceptable. Platform-only functions require Platform Admin privileges. | Phase 0-5 | Approved. Every API endpoint is private unless explicitly approved as public. Authenticated requests execute within the authenticated user's Organization context. Organization-private information must never be returned to another Organization. | Decided |
| ODR-010 | What warehouse employee authentication is approved? | Require warehouse employees to authenticate using at least two factors of knowledge or possession before performing warehouse operations that affect operational records. Approved examples include Employee ID + PIN, Employee ID + Badge, Employee ID + Password, Badge + PIN, or equivalent multi-factor workflow appropriate for the customer's operational environment. | Rely solely on company employee number; allow unauthenticated confirmation; grant warehouse employees administrative privileges. | Warehouse actions affect operational records and must be tied to authenticated identity, Organization context, permissions, and audit logs. Authentication and authorization remain separate. | Phase 3-4 | Approved. Warehouse employees may perform only warehouse-authorized workflows such as loading confirmation, inventory confirmation, manifest confirmation, return verification, warehouse documentation, and authorized printing. They do not receive administrative privileges. | Decided |
| ODR-011 | Which safety intelligence may become platform-global? | Follow the principle: "The platform shares knowledge, not customer operations." Never share Organization-specific operational data. Share only reviewed, approved, and sanitized truck-safety intelligence such as verified low-clearance bridges, truck restrictions, weight/height restrictions, commercial-vehicle road closures, construction hazards, hazardous intersections, dangerous turns, unsafe loading entrances, unsafe commercial-vehicle parking locations, verified hazard GPS coordinates, and approved sanitized hazard photos. | Keep all safety data private; share all hazard submissions immediately; allow Organizations to directly view each other's operational data. | Protects customer operations and tenant isolation while allowing verified safety knowledge to benefit all Organizations. | Phase 7 | Approved. Shared safety submissions must be reviewed by Platform Administration or an approved moderation workflow, sanitized where appropriate, and approved before becoming platform-global. No Organization may directly access another Organization's private operational data. | Decided |
| ODR-012 | What Google Maps storage/caching legal posture is approved? | Store company-owned GPS trails/events and avoid permanently storing Google route line/content unless legal approval is obtained. | Cache Google content permanently; avoid Google-derived content entirely. | Terms uncertainty can create legal and operational risk. | Phase 0 | Reclassified as Operational Verification. Rationale: governance has approved the platform posture of storing company-owned operational events instead of permanent Google content; remaining work is verifying provider terms, key restrictions, and any required legal/commercial approvals. | Operational Verification |
| ODR-013 | What data retention model applies across platform and Organization data? | Use category-based retention with separate Operational Retention, Analytical Retention, and Backup Retention policies. Platform Admin controls platform audit logs, security logs, Shared Safety Intelligence, platform-global reference data, low-clearance bridge data, and truck restriction reference data. Organizations may configure retention for Organization-owned operational data such as route history, delivery history, driver performance history, delivery photos, operational reports, analytics, and generated documents. | Use one global retention period; defer retention until production; let Organizations control platform audit/security records. | Category-based retention supports future legal, regulatory, contractual, and customer-specific requirements without schema redesign while keeping platform-wide reference and audit/security records under Platform Admin control. | Phase 0-2 | Approved. AI-related retention must preserve explainability while avoiding unnecessary long-term storage of intermediate inference artifacts. Operational, Analytical, and Backup retention are separate policies, not one setting. | Decided |
| ODR-014 | Which deployment environments are required? | Maintain separate Development, Staging, Pilot, and Production environments. Normal release path is Development to Staging to Pilot to Production. Each environment has independent configuration, secrets, databases, object storage, logging, monitoring, backups, and audit records. | Development plus Production only; combine Staging and Pilot; add environments later. | Separate environments reduce migration, restore, performance, security, and pilot-release risk. Production data must not be copied into Development except through an approved sanitized process. | Phase 0 | Approved. Emergency fixes may bypass Pilot only with explicit Platform Admin approval and documented justification. | Decided |
| ODR-015 | What rollback baseline must be recorded for deployments? | Every production deployment must create a rollback baseline containing application commit/version/build ID, database schema/migration/backup IDs, mobile version/build/EAS ID where applicable, deployment environment/timestamp/package/config version, object-storage config version, API version, and external dependency versions where applicable. | Record only Git SHA; rely only on Render history; defer rollback details until failure. | A deployment is not complete until rollback verification requirements are satisfied. Incomplete baselines make recovery slow or unreliable. | Phase 0 | Approved. Rollback documentation must include trigger conditions, decision authority, validation steps, recovery verification, and post-rollback review. | Decided |
| ODR-016 | Which test/demo data should be preserved or excluded from tenant backfill? | Exclude demo routes, test drivers, QR artifacts, and temporary pilot build artifacts from production tenant data unless explicitly approved. | Backfill all existing records; delete all test data immediately. | Backfilling test data pollutes analytics; deleting before backup risks losing useful evidence. | Phase 2-6 | Reclassified as Engineering Decision. Rationale: ODR-001 approved `Truck-Safe Routing Development` as the internal development/test Organization; specific cleanup and backfill mechanics are migration implementation work. | Engineering Decision |
| ODR-017 | What API versioning strategy is approved? | Introduce `/api/v1` during tenant enforcement while maintaining compatibility redirects or adapters where needed. | Keep unversioned APIs; immediate breaking version switch. | Versioning reduces mobile breakage risk during tenant migration. | Phase 5 | Reclassified as Engineering Decision. Rationale: ODR-009 approved Private by Default and Organization-scoped access; concrete API versioning and compatibility adapters are implementation design decisions. | Engineering Decision |
| ODR-018 | What object-storage strategy is approved for production binary assets? | Use an Object Storage abstraction layer and keep the architecture vendor-neutral. Support interchangeable providers such as Amazon S3, Cloudflare R2, Azure Blob Storage, and Google Cloud Storage. Store object identifiers, Organization ownership, metadata, permissions, checksums, MIME type, size, upload timestamps, retention policy, and lifecycle status in the database; store binary assets in object storage. | Permanently bind to one storage vendor; store binary assets in the operational database; use direct public access by default. | Vendor-neutral object storage avoids business-logic redesign when providers change and preserves Organization-scoped access control for uploaded assets. | Phase 0 | Approved. Every uploaded object belongs to exactly one Organization unless explicitly Platform Global. Shared Safety objects follow ODR-011. Objects use immutable internal identifiers; user filenames are not permanent identifiers; direct public access requires explicit approval. | Decided |
