# Backup Restore Rehearsal

Target: isolated PostgreSQL/PostGIS only.

Status: PASSED for non-production rehearsal.

Planned and executed validation path:

1. Source validation database.
2. Logical backup using PostgreSQL tooling.
3. Restore into a new empty validation database.
4. Re-run migrations/readiness validators.
5. Compare key table counts and PostGIS availability.

Rehearsal evidence:

- Source database: `tsr_rollout_source_1784342000`
- Restore database: `tsr_rollout_restore_1784342000`
- PostgreSQL version: 17.10
- Backup format: PostgreSQL custom dump via `pg_dump -Fc`
- Restored with: `pg_restore`
- Restored database preflight: passed
- PostGIS enabled after restore: true
- Migrations 001-008 present after restore: true
- Key table-count comparison from preflight: matched for Organizations, admin users, drivers, route manifests, Shared Safety, BI/KPI, Logistics Intelligence, and FISS tables.

What this proves:

- The current schema and data generated by non-production validators can be dumped and restored.
- Migrations 001-008 and PostGIS are compatible with restore validation in a local environment.
- Application readiness can be verified after restore.

What this does not prove:

- Actual production provider backup exists.
- Actual production provider restore works.
- Production point-in-time recovery is enabled.
- Production object storage can be restored.

Production backup/restore remains OPERATIONAL_VERIFICATION_REQUIRED because provider-managed production backup and restore evidence was not inspected.
